By separating online discovery functionality from the application's primary reading interfaces, BookSearchDrawer helps maintain a modular architecture where search, retrieval, and presentation responsibilities remain clearly isolated.
BookProvider defines the abstraction layer for all online book discovery sources by establishing a single asynchronous search contract that returns collections of OnlineBook results. BookProvider accomplishes this by requiring conforming providers to implement searchBooks(query:) and guaranteeing a consistent return type regardless of the underlying service, API, or data source being queried. BookProvider collaborates directly with OnlineBook and any concrete provider implementations, serving as the common interface that allows multiple discovery services to participate in the same search workflow. BookProvider enables higher-level components such as discovery coordinators, repositories, or view models to interact with online book sources without depending on provider-specific implementations. By standardizing how search operations are executed and consumed, BookProvider helps maintain a scalable and interchangeable architecture where additional book sources can be introduced without requiring changes throughout the rest of the application.
OpenLibraryProvider serves as the primary online discovery implementation by fulfilling the BookProvider contract and retrieving book data from the Open Library Search API. OpenLibraryProvider performs this responsibility by validating and encoding user queries, executing network requests through URLSession, decoding Open Library search responses into OpenLibrarySearchResponse and OpenLibraryDocument models, and transforming API records into standardized OnlineBook instances through mapDocumentToOnlineBook. OpenLibraryProvider collaborates directly with BookProvider, OnlineBook, URLSession, OpenLibrarySearchResponse, and OpenLibraryDocument, acting as the translation layer between an external book catalog service and the application's internal discovery model. OpenLibraryProvider enables BookDiscoveryViewModel and other discovery-related components to obtain normalized search results without needing knowledge of Open Library's API structure, response schema, or networking requirements. By encapsulating external communication, response parsing, data validation, and model conversion within a single provider, OpenLibraryProvider helps keep the broader discovery architecture provider-agnostic, maintainable, and easily extensible for additional book sources.
GutenbergProvider functions as an additional online discovery source by implementing the BookProvider contract and retrieving public-domain book data from the Gutendex API, which exposes the Project Gutenberg catalog. GutenbergProvider achieves this by validating search input, performing asynchronous network requests through URLSession, decoding API responses into GutendexResponse, GutendexBook, GutendexAuthor, and GutendexFormats models, and converting those external records into standardized OnlineBook instances through mapBook. GutenbergProvider collaborates directly with BookProvider, OnlineBook, URLSession, and its internal API decoding models, serving as a normalization layer between the Gutendex service and the application's discovery infrastructure. GutenbergProvider enables BookDiscoveryViewModel and other discovery-related systems to surface freely accessible books alongside results from other providers without needing to understand Gutendex-specific response formats or networking behavior. By contributing a source of downloadable public-domain content while conforming to the same abstraction as other providers, GutenbergProvider helps expand discovery coverage and supports a provider-agnostic search architecture throughout the application.
ContentView functions as the primary application shell for the reading experience, coordinating book library management, reader navigation, import workflows, deletion operations, and the presentation of reading-time predictions generated from user interaction. ContentView accomplishes this by consuming DataStore and SessionCoordinator environment objects, rendering the library through NavigationSplitView, launching PDFReaderScreen or EPUBReaderScreen based on Book metadata, importing content through BookImporter, and invoking EstimationEngine after sustained hover events to generate BookEstimationResult data for the detail pane. ContentView collaborates directly with DataStore, SessionCoordinator, Book, BookImporter, EstimationEngine, BookEstimationResult, PDFReaderScreen, EPUBReaderScreen, ContentBookRowView, and BookEstimationView, serving as the orchestration layer that connects reading, importing, estimation, and navigation systems into a unified interface. ContentBookRowView supports this architecture by providing a reusable visual representation of library entries, while BookEstimationView transforms BookEstimationResult data into a detailed prediction interface containing completion forecasts, session expectations, chapter-level estimates, and confidence indicators. Together, ContentView, ContentBookRowView, and BookEstimationView enable the application's core value proposition by connecting stored reading data to estimation intelligence, allowing users to discover, open, manage, and evaluate reading commitments from a single centralized workflow.
